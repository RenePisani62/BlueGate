# ADR-0004: Persist and De-duplicate Alerts Using SQLite

## Status

Accepted

## Context

BlueGate initially displayed detections only in the console. Alerts disappeared when the application exited, preventing historical investigation, reporting and future dashboard functionality.

Because BlueGate currently rereads a limited window of recent Sysmon events on every execution, the same Windows event may be analysed more than once. Each generated alert receives a new GUID, so using only the alert ID cannot prevent logical duplicates.

## Decision

BlueGate will persist alerts locally using SQLite through an `AlertRepository`.

The repository is responsible for:

* creating the alert database;
* inserting newly generated alerts;
* retrieving stored alerts;
* isolating database-specific logic from the detection engine and presentation layer.

Each `SysmonNetworkEvent` will retain its Windows Event Log Record ID.

The `Alerts` table will enforce this composite uniqueness rule:

`EventRecordId + RuleName`

This permits one Sysmon event to match several different detection rules while preventing the same rule from storing the same source event repeatedly.

The repository `Save()` method will return:

* `true` when a new row is inserted;
* `false` when SQLite ignores a duplicate.

## Rationale

SQLite was selected because it:

* operates locally without a separate database service;
* is lightweight and appropriate for an endpoint agent;
* supports transactions, indexes and uniqueness constraints;
* provides a practical foundation for the future dashboard;
* can later be replaced or supplemented by centralised storage behind the repository abstraction.

## Consequences

### Advantages

* Alerts survive application restarts.
* BlueGate can distinguish current detections from historical alerts.
* Reprocessing a Sysmon event does not create repeated database rows for the same rule.
* Database concerns remain outside the detection engine.
* Future interfaces can query the repository without depending directly on SQLite.
* Historical investigation, timelines and reporting become possible.

### Limitations

* BlueGate currently reads only a recent snapshot of Sysmon events.
* The development database is stored beneath the build-output directory.
* Existing schema changes currently require recreating the development database.
* Retrieved alerts do not yet reconstruct every original event property.
* A future Windows Service will require a durable location such as:

`C:\ProgramData\BlueGate\Data\bluegate.db`

## Future Work

* Add schema migration support.
* Store the database in a durable application-data path.
* Add repository filtering and paging.
* Reconstruct full event objects when reading alerts.
* Add indexes for timestamp, severity, rule and MITRE technique.
* Support centralised multi-host storage in a later BlueGate architecture.
