# ADR-010: HealthMonitor Uses AlertRepository for Database Connectivity Checks

## Status

Accepted

## Date

17 July 2026

## Context

Added Health Monitor which equips BlueGate to monitor its own health in regards to database and sysmon connections.

BlueGate's health dashboard initially reported database connectivity using a
hard-coded value. A genuine runtime check was required so that the agent could
verify whether its SQLite database was accessible.

HealthMonitor could open a SQLite connection directly, but that would duplicate
database-access knowledge outside the repository layer.

## Decision

AlertRepository will expose a TestConnection() method.

HealthMonitor will receive AlertRepository through constructor injection and
call TestConnection() when creating HealthStatus.

HealthMonitor will not directly reference Microsoft.Data.Sqlite or construct a
database connection.

## Consequences

### Positive

- Database access remains encapsulated within the repository layer.
- The dashboard reports verified runtime health rather than a placeholder.
- HealthMonitor is not coupled directly to SQLite.
- The dependency is explicit and can later be replaced with an interface for
  testing or an alternative persistence implementation.

### Negative

- HealthMonitor is currently coupled to the concrete AlertRepository class.
- A database connection is opened during each health check.
- Additional constructor wiring is required in BlueGateAgent.

## Future Considerations

- Introduce an interface such as IAlertRepository or IDatabaseHealthCheck.
- Consider reducing the frequency of database health checks if polling becomes
  expensive.
- Return richer health-check results containing failure details and duration.
