# ADR-7: Introduce Dependency Injection for Health Status Provider

## Status

Accepted

## Context

The initial implementation of the BlueGate dashboard created a HealthStatus object directly inside the Razor PageModel (IndexModel). Although functional, this tightly coupled the presentation layer to the construction of health data.

As BlueGate evolves, health information may originate from multiple sources including:

- BlueGate Agent
- SQLite
- Windows APIs
- Remote agents
- REST APIs

The web layer should not know how this information is obtained.

## Decision

Introduce an interface:

    IHealthStatusProvider

implemented by:

    HealthStatusProvider

Register the implementation with ASP.NET Core's Dependency Injection container using:

builder.Services.AddSingleton<IHealthStatusProvider, HealthStatusProvider>();

Inject the dependency into IndexModel through constructor injection.

The PageModel now requests health information via:

Health = _healthStatusProvider.GetStatus();

instead of constructing HealthStatus directly.

## Consequences

Advantages

- Separation of concerns.
- Presentation layer no longer owns health object creation.
- Enables replacement of the provider without changing the Razor Page.
- Establishes a reusable pattern for future services.

Disadvantages

- Slight increase in project complexity.
- Requires understanding of Dependency Injection and service registration.

## Future Direction

Replace the temporary HealthStatusProvider implementation with a provider that communicates with the running BlueGate Agent and retrieves live monitoring information.
