# ADR-0006

## Title

Continuous Monitoring Using Incremental Sysmon Event Processing

## Status

Accepted

## Context

BlueGate initially operated as a console application that processed a fixed number of recent Sysmon events before terminating.

While suitable for early development, this approach repeatedly analysed historical events and required manual execution.

A continuously running monitoring agent requires incremental processing of newly generated telemetry.

---

## Decision

BlueGate shall operate as a continuously running monitoring agent.

The application:

- Maintains an in-memory record of the highest processed Sysmon Event Record ID.
- Processes only Sysmon Event ID 3 records created after that point.
- Executes processing through `RunOnce()`.
- Manages application lifetime through `Run()`.

Application startup responsibility remains within `Program.cs`, while operational responsibility resides within `BlueGateAgent`.

---

## Consequences

### Advantages

- Reduced CPU utilisation.
- Reduced memory allocations.
- Elimination of duplicate processing.
- Improved scalability.
- Foundation for Windows Service implementation.
- Clear separation between application lifetime and processing.

### Disadvantages

- State currently exists only in memory.
- A restart resets the last processed Event Record ID.
- Persistent checkpoint storage will be implemented in a future iteration.

---

## Future Work

- Persist Event Record ID to SQLite.
- Replace polling with event-driven processing where practical.
- Implement graceful shutdown.
- Add performance metrics.
