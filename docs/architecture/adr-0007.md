ADR-007

Title

Persistent Configuration Repository

Status

Accepted

Context

BlueGate previously maintained application state only in memory.

Following a restart or unexpected termination, monitoring resumed from the latest Sysmon event, preventing continuity and future checkpoint recovery.

A persistent configuration mechanism was required to support:

application settings
monitoring checkpoints
future runtime configuration
Decision

Implement a dedicated ConfigurationRepository using SQLite.

The repository provides:

Initialise()
GetValue()
SetValue()

Configuration is stored using a generic key/value model.

SQLite UPSERT is used to simplify updates while maintaining a single SQL statement.

Rationale

This approach:

separates configuration from alerts
supports future expansion without schema redesign
keeps persistence logic isolated
aligns with the Single Responsibility Principle
supports future migration to alternative database engines
Consequences
Positive
Persistent application state
Restart resilience
Simplified configuration management
Extensible architecture
Clean separation of responsibilities
Secure parameterised SQL
Negative
Slight increase in architectural complexity
Additional repository dependency within BlueGateAgent
Future Work
Complete checkpoint loading during startup.
Persist latest processed Event Record ID after each monitoring cycle.
Add additional configuration keys:
Polling interval
Log level
Detection rule configuration
Dashboard preferences
Introduce typed configuration objects.
Investigate dependency injection for repository creation.
