# ADR-0008: Persist Monitoring Checkpoints and Resume Incrementally

## Status

Accepted

## Context

BlueGate continuously processes Sysmon Event ID 3 network telemetry.

An in-memory checkpoint prevented duplicate processing during a single execution, but that state was lost whenever BlueGate stopped or Windows restarted.

Without persistent checkpoint recovery, BlueGate could:

- miss events created while the agent was offline;
- recreate a baseline on every startup;
- reprocess historical telemetry;
- produce inconsistent monitoring continuity.

## Decision

BlueGate will persist the highest successfully processed Sysmon Event Record ID in SQLite.

The checkpoint is stored through `ConfigurationRepository` under:

`LastProcessedEventRecordId`

At startup:

- if a checkpoint exists, BlueGate resumes after it;
- if no checkpoint exists, BlueGate creates a baseline using the latest available Sysmon network Event Record ID.

During runtime:

- BlueGate reads only Event ID 3 records newer than the active checkpoint;
- it processes events in batches;
- it persists the updated checkpoint only after the batch has been analysed and alert persistence has completed.

## Rationale

Persisting the checkpoint provides continuity across:

- normal application restarts;
- Windows reboots;
- unexpected process termination;
- temporary monitoring interruptions.

Updating the checkpoint after successful processing favours at-least-once processing. Reprocessing a batch is preferable to permanently skipping unprocessed events.

Duplicate alert persistence is mitigated through the existing composite uniqueness rule:

`EventRecordId + RuleName`

## Consequences

### Positive

- Monitoring resumes from the last confirmed position.
- Historical events are not repeatedly processed.
- Events produced while BlueGate is offline can be processed after restart.
- Runtime processing is more efficient.
- The agent behaves more like a production monitoring service.
- Checkpoint state is transparent and inspectable.

### Negative

- Checkpoint persistence introduces additional database writes.
- The current implementation stores checkpoint state in the generic configuration table.
- Processing is currently limited to fixed-size batches.
- A large backlog may require several cycles to catch up.
- Multiple future telemetry sources will require independent checkpoints.

## Operational Behaviour

```text

  ↓
Load checkpoint
  ↓
Checkpoint exists?
  ├── Yes → resume after stored record
  └── No  → create baseline from latest Sysmon record

  ↓

  ↓
