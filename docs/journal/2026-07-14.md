Development Journal – Day 9

Project: BlueGate – Windows SIEM / Endpoint Monitoring Agent

Development Time: ~4 hours (estimated)

Objectives

Today's goal was to complete the ConfigurationRepository and begin integrating persistent application state into the BlueGate architecture.

Rather than simply adding database functionality, the focus was on understanding object responsibilities, application lifecycle, and clean software architecture.

Technical Work Completed
Completed ConfigurationRepository

Implemented the three core repository methods:

Initialise()
Creates the Configuration table if it does not already exist.
Uses idempotent initialisation (CREATE TABLE IF NOT EXISTS).
GetValue()
Retrieves configuration values using parameterised SQL.
Returns null when a configuration key does not exist.
Uses ExecuteScalar() appropriately.
SetValue()
Implements SQLite UPSERT functionality.
Automatically inserts or updates configuration values.
Uses parameterised SQL to prevent SQL injection.
BlueGateAgent Integration
Added ConfigurationRepository as a new dependency.
Instantiated the repository during object construction.
Initialised the repository during BlueGate startup.

(Checkpoint loading logic is partially prepared and will be completed next session.)

Engineering Concepts Learned

Today's session focused heavily on software engineering rather than simply writing code.

Constructors

Learned that constructors:

execute automatically when an object is created
prepare the object for use
are the correct location for creating long-lived dependencies

Analogy:

The builder arrives on site, assembles the team, and prepares everything before construction begins.

Object Lifetime

Reinforced understanding of:

object scope
using var
automatic disposal
resource lifetime
Separation of Responsibilities

Reinforced that:

repositories manage persistence
BlueGateAgent coordinates components
SQLite manages SQL execution

Each layer performs only the work for which it is responsible.

Persistent vs Volatile State

Discussed:

runtime memory
persistent storage
checkpoint recovery
baseline creation

BlueGate is now evolving from a stateless application into one capable of surviving restarts.

Secure Coding

Revisited parameterised SQL queries and discussed:

SQL Injection
Secure by Design
avoiding string concatenation

This reinforced previous Master's coursework within a practical implementation.

UPSERT

Introduced SQLite UPSERT.

Rather than BlueGate deciding between INSERT or UPDATE, SQLite performs the decision internally.

This reinforced the principle:

Let each component perform the work it was designed to perform.

Design Decisions

Agreed that:

Configuration should generally be read once during startup.
Checkpoints should be written after each monitoring cycle.
Future monitoring sources (Registry, Defender, Services, etc.) should each maintain independent checkpoints.
Reflections

Today's session shifted further away from learning C# syntax and toward software engineering principles.

The project is becoming a vehicle for learning architecture, object-oriented design, state management and clean application structure while simultaneously developing a commercial-quality security product.
