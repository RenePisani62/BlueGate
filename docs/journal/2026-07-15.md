# BlueGate Development Journal — 15 July 2026

## Objectives

- Complete persistent checkpoint recovery.
- Verify continuous incremental Sysmon processing.
- Confirm the end-to-end detection and persistence pipeline.
- Investigate discrepancies between current-cycle alert counts and historical database totals.

## Work Completed

### Persistent Checkpoint Recovery

Integrated `ConfigurationRepository` into `BlueGateAgent`.

BlueGate now:

- reads `LastProcessedEventRecordId` during startup;
- creates a baseline from the latest Sysmon record when no checkpoint exists;
- stores the baseline in SQLite;
- resumes from the stored checkpoint on later executions;
- advances the checkpoint after successfully processing each event batch.

### Continuous Incremental Processing

Verified that BlueGate processes only Sysmon Event ID 3 records newer than the stored checkpoint.

Observed behaviour:

- previously stored checkpoints are loaded successfully;
- new events are processed in batches;
- the checkpoint advances to the highest processed Event Record ID;
- later cycles do not reprocess the same events.

### End-to-End Detection Test

Verified the complete pipeline:

1. Sysmon generated network connection events.
2. `SysmonReader` retrieved the events.
3. `DetectionEngine` matched PowerShell activity.
4. Alerts were enriched with MITRE ATT&CK context.
5. `AlertRepository` persisted alerts to SQLite.
6. Historical alerts were read back and displayed.
7. The checkpoint advanced after successful processing.

A batch of 100 Sysmon network events produced four PowerShell outbound-connection alerts.

The matching Sysmon Event Record IDs were:

- 920527
- 920533
- 920534
- 920716

All four events involved:

- `powershell.exe`
- the same user account;
- destination `23.211.130.78:443`.

### Alert Count Investigation

Clarified the distinction between:

- alerts generated during the current monitoring cycle;
- total alerts already stored in SQLite.

`Generated 0 alert(s) this run` refers only to the current cycle.

`Total Alerts` refers to all historical database records.

Further testing confirmed that earlier apparent inconsistencies were caused by observing different monitoring cycles and by batch processing of pending Sysmon events.

### Event Record ID Behaviour

Confirmed that Sysmon Event Record IDs are shared across the entire Sysmon operational log, not only Event ID 3.

Therefore, processing 100 network events may advance the checkpoint by more than 100 record numbers because other Sysmon event types occur between network events.
