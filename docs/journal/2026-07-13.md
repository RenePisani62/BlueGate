# Development Journal – 13 July 2026

## Objective

Continue transforming BlueGate from a console utility into a continuously running endpoint monitoring agent.

---

## Work Completed

### BlueGateAgent Refactoring

- Introduced the `BlueGateAgent` class to encapsulate application orchestration.
- Reduced `Program.cs` to a minimal application entry point.
- Separated application lifetime management from processing logic.
- Added `Run()` and `RunOnce()` methods to clearly separate continuous execution from a single monitoring cycle.

---

### Continuous Monitoring

Implemented the first continuously running monitoring loop.

The agent now:

- Starts once
- Monitors Sysmon every five seconds
- Processes newly detected events
- Persists alerts
- Waits before beginning the next cycle

This establishes the foundation for a future Windows Service.

---

### Incremental Event Processing

Implemented tracking of the most recent Sysmon Event Record ID processed.

BlueGate now:

- Establishes a baseline Event Record ID when starting.
- Processes only newer Sysmon Event ID 3 entries.
- Avoids reprocessing historical events during runtime.
- Reduces CPU utilisation and duplicate processing.

---

### Runtime Testing

Performed multiple runtime tests.

Verified:

- Idle monitoring generates no alerts.
- New PowerShell outbound connections are detected.
- MITRE ATT&CK enrichment remains functional.
- Alerts are persisted to SQLite.
- Historical alerts remain available across application restarts.
- Duplicate alerts are prevented during continuous execution.

---

## Architectural Progress

BlueGate now behaves as a continuously running endpoint monitoring agent rather than a single-run console application.

Current processing pipeline:

Sysmon
→ Detection Engine
→ MITRE Enrichment
→ SQLite Repository
→ Console Presentation

The architecture now supports future implementation as a Windows Service.

---

## Lessons Learned

Separating application lifetime (`Run`) from processing (`RunOnce`) greatly simplifies future enhancements.

Maintaining the last processed Event Record ID significantly reduces unnecessary processing and forms the basis of an efficient monitoring agent.
